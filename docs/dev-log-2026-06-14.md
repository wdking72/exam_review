## 开发了什么

### 1. Keyword Search / BM25

```
src/rag/keyword-search.ts — KeywordSearch 类
```

**倒排索引**：`Map<term, Map<docIndex, count>>` — 从词到文档的快速查找。

**BM25 评分公式**：
- IDF(t) = ln((N - df + 0.5) / (df + 0.5) + 1) — 稀有词权重更高
- TF 归一化 = (tf × (k1+1)) / (tf + k1 × (1 - b + b × docLen/avgDocLen)) — 抑制长文档
- 参数 k1=1.5（TF 饱和度），b=0.75（长度归一化强度）

### 2. Reranker

```
src/rag/reranker.ts — rerank() 函数
```

向量分和 BM25 分分量纲不同，各自归一化到 [0,1] 后融合：

`finalScore = alpha × vecNorm + (1-alpha) × kwNorm`

以 text 为 key 合并两边结果，只有一边命中的补 0。

### 3. Query Rewriter

```
src/rag/query-rewriter.ts — QueryRewriter 类
```

用 LLM 改写用户查询：代词→具体名词、补全省略、去掉口语化。LLM 返回空时 fallback 到原问题。

### 4. Hybrid Search 流水线

```
src/rag/rag-engine.ts — 改进
```

三阶段流水线：

```
用户查询 → QueryRewriter 改写
         → 向量检索 (topK×3) + 关键词检索 (topK×3)
         → Reranker 融合重排 → 取 Top-K
```

KeywordSearch 和 QueryRewriter 均为可选注入，不启用时退化为纯向量检索。

### 5. Embedding Cache

```
src/rag/rag-cache.ts — 缓存 Embedding 结果
```

**解决的问题**：加载知识库时要 Embedding API 调用，每次启动都重复计算。

**方案**：将 `texts → embeddings` 的映射缓存到 `.rag-cache/` 目录。启动时检测缓存是否匹配（文件未变、chunk 数一致），命中则直接加载缓存，跳过 API 调用。

### 6. 类型重构

```
原来：src/types.ts
现在：src/types/index.ts + src/types/rag.ts
```

将 RAG 相关类型（`RerankResult` 等）抽到独立文件，核心类型留在 `index.ts`。

### 7. GitHub Actions CI

`.github/workflows/` — 提交时自动 `npm ci` + `tsc --noEmit` 类型检查。

## 为什么开发这些功能

- **纯向量检索的局限**：语义相似 ≠ 关键词匹配，专有名词（如"微分方程"）在向量检索中容易丢失精度
- **Hybrid Search**：向量 + 关键词互补，向量覆盖语义，BM25 覆盖精确匹配
- **Reranker**：不同检索方式的得分不能直接比较，需要归一化融合
- **Query Rewriter**：用户输入常含糊（"它"、"这个"），改写后检索效果更好
- **Embedding Cache**：每次启动都重新 embedding 太慢、费钱

## 踩坑记录

### 1. BM25 IDF 公式写错

`Math.log(N - df + 0.5) / (df + 0.5 + 1)` 是错的，正确是 `Math.log((N - df + 0.5) / (df + 0.5) + 1)`。

### 2. VectorStore.add() 签名

`add(text, vector, metadata)` 只有 3 个参数，不要传 id。

### 3. const 数组 + push

`const arr = []` 不能重新赋值但可以 push，不要混淆。

### 4. 数据同步问题

加载知识库时要同步维护向量库和关键词索引，两边的数据必须一致。

### 5. 缓存一致性

文件内容变更后缓存必须失效。通过对比 `texts.length` 做快速校验。
