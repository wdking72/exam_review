# 开发日志 — 2026-06-12

## 概述

核心 Agent 框架开发。从零搭建了 ReAct Agent 循环、Tool 抽象、Tool Registry，并接入真实 LLM（Claude API 和 OpenAI 兼容 API）。

---

## 开发了什么

### 1. ReAct Agent 核心（Stage 1）

```
src/
├── core/agent.ts          # Agent 类 — ReAct 循环实现
├── types.ts               # ToolCall、AgentStep、Tool 等核心类型
├── llm/llm.ts             # LLM 接口 + MockLLM
└── tools/tools.ts         # mockKnowledgeBase、generatePractice 工具
```

**Agent 类**：经典的 ReAct（Reason + Act）循环，LLM 每轮输出 `Thought → Action → Observation` 或直接 `Answer`，循环直到拿到最终答案或超出最大迭代次数。

### 2. Tool Registry（Stage 2）

集中管理所有工具的注册中心，统一执行入口做参数校验。

### 3. 接入真实 LLM（Stage 3、Stage 4）

| 文件 | 说明 |
|------|------|
| `llm/llm-claude.ts` | ClaudeLLM — 封装 Anthropic SDK 实现 LLM 接口 |
| `llm/llm-openai.ts` | OpenAICompatibleLLM — 兼容硅基流动等第三方 API |
| `core/agent-native.ts` | NativeToolAgent — 使用 API 原生 `tools` 参数，无需文本解析 |

两种 Agent 模式：`ReAct Prompting`（文本解析）和 `Native Tool Use`（结构化输出）。

### 4. 初始工具

- `search_knowledge_base` — 搜索教材笔记（mock 版，硬编码 Q&A）
- `generate_practice` — 生成模拟练习题

---

## 为什么开发这些功能

- **核心需求**：Agent 不是简单的 LLM 问答，需要「思考→行动→观察」的循环才能调用工具解决复杂问题
- **Tool Registry**：多个工具需要统一管理和校验，不能硬编码
- **Native Tool Use**：比文本解析更可靠，LLM 返回结构化 JSON 而非自由文本

## 踩坑记录

### 1. 正则 `$` 锚点误用

`/Action:\s*(\w+)$(\{.*?\})/` 中的 `$` 要求匹配行尾，Action 永远不会被匹配。改为 `\(` 匹配左括号。

### 2. MockLLM 一次性返回

Mock 在同一轮同时返回 Action 和 Answer，ReAct 循环形同虚设。修复为按阶段返回：第一轮只给 Action，收到 Observation 后才给 Answer。

### 3. `!args[param]` 误判空字符串

参数校验用 `!args[param]` 时，空字符串 `""` 被视为 falsy。改为 `!(param in args)` 只检查 key 是否存在。

### 4. 环境变量 typo

`ANROPIC_API_KEY` 少写了一个 H，正确的环境变量名是 `ANTHROPIC_API_KEY`。

### 5. Native Tool Use 模型兼容

不是所有 OpenAI 兼容模型都真正支持 `tools` 参数的结构化输出。`Nex-N2-Pro` 接受参数但无视，`Qwen2.5-7B` 调用格式错乱。

### 6. tool_call_id 必须匹配

tool 角色的消息必须带上正确的 `tool_call_id`，否则 API 报错。
