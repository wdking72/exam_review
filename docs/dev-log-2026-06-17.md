### 5. 流式对话功能完善（useChat.ts）

重写 `useChat` composable，核心改动：

**SSE 流式通信原理（详细注释）：**

```
前端 (useChat)  →  POST /api/chat/stream  →  后端 (Koa + Agent)
     ↑                                              │
     └──────────── SSE (Server-Sent Events) ────────┘
```

- 为什么用 SSE 而不是 WebSocket：SSE 是单向推送（服务端→客户端），正好满足 LLM 流式输出场景；基于 HTTP，天然支持 CORS、代理
- 为什么用 fetch ReadableStream 而不是 EventSource：EventSource 只支持 GET，无法 POST JSON body

**处理流程：**

1. fetch() POST 请求，获取 ReadableStream
2. reader.read() 循环读取二进制 chunk
3. TextDecoder 解码→buffer 缓冲→按 "\n" 分割→逐行解析 "data:" 前缀
4. 根据 chunk.type（text/tool/done/error）追加到 UI 消息
5. 收到 done 或流结束时退出循环

**新增功能：**

- `cancelMessage()`：通过 AbortController 支持取消进行中的流式请求
- 自动滚动：每收到文本 token 都 scrollToBottom()
- 错误处理：区分 AbortError（用户取消）和其他错误
- 类型定义：StreamChunk 接口规范化后端推送的数据结构

**ChatView.vue 配套更新：**

- 加载状态新增"停止"按钮，调用 cancelMessage()
- 解构出 cancelMessage

### 6. SSE 流式输出 Bug 排查与修复（重点）

**现象**：前端发送问题后，助手回答不是逐字出现，而是等整个回答生成完后一次性整段打印出来。

**排查思路（假设驱动）**：

| 假设 | 内容                                                                                | 验证结果                                                                                    |
| ---- | ----------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------- |
| A    | 后端 `sendSSEEvent()` 只 `write()` 不 `flush()`，数据积压在 Node.js / OS TCP 缓冲区 | **Rejected**：curl 直连后端得到 86 个 chunk，后端确实在流式发送                             |
| B    | Vite 开发服务器代理缓冲了 SSE 响应                                                  | **Confirmed**：通过 Vite 代理时前端只收到 1 个 3596B 的大 chunk；直连后端则收到 86 个 chunk |
| C    | LLM 上游（OpenAI SDK / 模型服务）本身没有真正流式返回                               | **Rejected**：调试日志显示 `agent-native.ts` 的 onToken 回调逐 token 触发                   |
| D    | 前端 Vue 响应式更新有批次合并，导致视觉上“一次性出现”                               | **Rejected**：前端收到的就是单一大 chunk，问题在到达前端之前                                |

**取证过程**：

1. **启动 Debug Server**：由于 Python 调试服务器在当前沙箱无法启动，使用 Node.js 实现了兼容的 fallback debug server，监听 `127.0.0.1:7777/event`，统一收集前后端日志。
2. **代码插桩**：在不改动业务逻辑的前提下，在 3 个关键位置插入日志上报：
   - `src/core/agent-native.ts`：每个 LLM text token 触发时上报（验证假设 C）
   - `src/server/routes/chat.ts`：每次调用 `sendSSEEvent()` 时上报（验证假设 A）
   - `frontend/src/composables/useChat.ts`：每次 `reader.read()` 拿到原始 chunk 和解析出 text token 时上报（验证假设 B/D）
3. **对比实验**：
   - **实验 1**：`node .dbg/test-sse.cjs` 直连 `http://localhost:3001/api/chat/stream`
     - 结果：86 个 chunk，耗时约 16s，后端/LLM 流式正常。
   - **实验 2**：浏览器通过 `http://localhost:5173`（Vite 代理）访问 `/api/chat/stream`
     - 结果：前端只收到 1 个 3596B 的 raw chunk，Vite 代理缓冲实锤。

**根因**：

1. **Vite 开发服务器内置的 http-proxy 会缓冲 SSE 响应**，等后端请求完全结束后才一次性推送给浏览器。
2. **后端 `createSSEStream()` 没有显式设置 `ctx.status = 200`**，直接 `flushHeaders()` 会把 Koa 默认的 404 状态发出去，导致浏览器 fetch 收到 404（虽然响应体仍在流式推送）。

**修复方案**：

| 文件                                  | 修改                                                                                           |
| ------------------------------------- | ---------------------------------------------------------------------------------------------- |
| `frontend/src/composables/useChat.ts` | 开发环境直连 `http://127.0.0.1:3001/api/chat/stream`，生产环境仍走 `/api/chat/stream`          |
| `frontend/vite.config.ts`             | 代理目标改为 `http://127.0.0.1:3001`，避免 Windows 下 `localhost` 解析到 IPv6 ::1 导致连接问题 |
| `src/server/sse.ts`                   | `createSSEStream()` 中显式 `ctx.status = 200` 后再 `flushHeaders()`                            |

**验证结果**：

- 修复前：前端通过 Vite 代理收到 1 个 3596B 大 chunk，视觉上整段出现。
- 修复后：前端直连后端，收到 60+ 个 chunk，HTTP 200，浏览器中助手消息逐字流式输出。
- TypeScript 检查通过，已移除所有临时调试插桩代码，后端用无插桩代码重启。

**经验教训**：

- SSE 问题要“分段排查”：先 curl 直连后端确认后端真流式，再查代理/中间件，最后查前端。
- Vite 开发代理虽然方便，但对 SSE 不友好，开发环境最好直连后端或配置专门代理插件。
- Koa 中直接操作 `ctx.res` 时，一定要记得显式设置 `ctx.status = 200`，否则默认 404 会坑掉 fetch。

## 踩坑记录

### 1. `@apply bg-background` 在 `@layer base` 中报错

**问题**：PostCSS 报错 `The 'bg-background' class does not exist`，即使 Tailwind 配置中已定义 `background` 颜色。

**原因**：`@layer base` 中使用 `@apply` 依赖 Tailwind 的 utilities 生成顺序，自定义颜色在 base 层可能尚未注册。

**解决**：改为直接写 CSS 属性 `background-color: #FAFAFA; color: #1F2937;`，避免 `@apply` 依赖。

### 2. Vite 开发服务器端口冲突

**问题**：`npm run dev` 时报 `Port 5173 is in use, trying another one...`，实际绑定到 5174，但 `agent-browser` 连接被拒。

**原因**：之前多次启动 Vite 进程未正确退出，残留 TIME_WAIT 连接。

**解决**：通过 `taskkill /PID <pid> /F` 杀掉残留进程；改用 `npx vite --port 5175 --host` 指定端口。根本解决需确保旧进程完全退出。

### 3. SSE 流式输出变成“整段出现”

**问题**：助手回答不是逐字流式打印，而是等全部生成完后一次性整段出现。

**原因**：

1. Vite 开发服务器内置的 http-proxy 会缓冲 SSE 响应，等后端请求结束后才一次性推送给浏览器。
2. 后端 `createSSEStream()` 未设置 `ctx.status = 200`，直接 `flushHeaders()` 导致浏览器收到 404。

**排查方法**：

- 先用 `curl` / Node.js 脚本直连后端，确认后端是否真的流式发送（我们得到 86 个 chunk，证明后端正常）。
- 再对比浏览器通过 Vite 代理的请求，发现只收到 1 个 3596B 大 chunk，定位到代理缓冲。
- 在 LLM 回调、`sendSSEEvent()`、前端 `reader.read()` 三处插桩，用 Debug Server 统一收集日志，排除 LLM 上游和前端消费逻辑的问题。

**解决**：

- 开发环境前端直连 `http://127.0.0.1:3001/api/chat/stream`，绕过 Vite 代理。
- `createSSEStream()` 中显式 `ctx.status = 200` 后再 `flushHeaders()`。
- `vite.config.ts` 代理目标改用 `127.0.0.1`，避免 `localhost` IPv6 解析问题。
