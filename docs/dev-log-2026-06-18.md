## 开发了什么

### 1. SSE 流式延迟 Bug 排查与修复

**现象**：用户发送“大苏打”后，AI 回复只有寥寥数十字，但 Network 面板显示总耗时约 43 秒，其中 `Content Download` 占 42.76 秒；用户感知到的流式打印仅 2-3 秒。

**排查思路（假设驱动）**：

| 假设 | 内容                                                              | 验证结果                                                 |
| ---- | ----------------------------------------------------------------- | -------------------------------------------------------- |
| A    | 服务端/反向代理对 SSE 响应做了缓冲，token 没有实时 flush 到客户端 | **可能**：`sendSSEEvent()` 只 `res.write()` 无显式 flush |
| B    | 大模型生成前的前置流程（RAG、工具调用、安全检查）阻塞了首 token   | **未复现**：本次测试无 tool_start 事件                   |
| C    | 服务端生成结束后未正确关闭流，连接空等到超时                      | **未复现**：流正常结束                                   |
| D    | Network 面板统计的 42 秒请求并非流式请求，而是其他后台请求        | **待排除**：需更多现场信息                               |
| E    | 前端流式消费逻辑存在处理延迟                                      | **Rejected**：前端收到数据后立即渲染                     |

**取证过程**：

1. **启动 Debug Server**：当前环境无 Python，使用 Node.js 实现兼容的 fallback debug server，监听 `127.0.0.1:7777/event`，统一收集前后端日志。
2. **代码插桩**：在不改动业务逻辑的前提下，在 3 处插入日志：
   - `src/core/agent-native.ts`：记录 LLM 请求启动、首 token TTFB、每个 text token 到达
   - `src/server/sse.ts`：记录响应头 flush、每次 SSE 写入
   - `frontend/src/composables/useChat.ts`：记录请求启动、每次前端收到 token
3. **复现结果**：
   - 请求启动 → 响应头 flush：36 ms
   - 请求启动 → LLM 首 token：~2.3 s
   - LLM 首 token → 前端收到：3 ms
   - **结论**：本次本地 dev 复现未出现 40 秒延迟，SSE 数据在生成后几毫秒内到达前端。

**根因分析**：

虽然本次未稳定复现 40 秒延迟，但 `src/server/sse.ts` 的 `sendSSEEvent()` 存在明确隐患：

```ts
export function sendSSEEvent(res: ServerResponse, event: string, data: string) {
  res.write(`event: ${event}\ndata: ${data}\n\n`);
}
```

- 仅调用 `res.write()`，没有显式 flush；在启用压缩中间件或某些 TCP/代理条件下，SSE 帧可能被缓冲，导致客户端“整段出现”。
- 未禁用响应压缩，压缩缓冲也可能导致 SSE 不实时。

**修复方案**：

| 文件                | 修改                                                              |
| ------------------- | ----------------------------------------------------------------- |
| `src/server/sse.ts` | `sendSSEEvent()` 每次 `res.write()` 后加 `(res as any).flush?.()` |
| `src/server/sse.ts` | SSE 响应头增加 `Content-Encoding: identity`，禁止压缩缓冲         |

修复后代码：

```ts
export function sendSSEEvent(res: ServerResponse, event: string, data: string) {
  res.write(`event: ${event}\ndata: ${data}\n\n`);
  (res as any).flush?.();
}
```

**验证结果**：

- 用户反馈修复后流式输出**变快了**。
- `npm run typecheck` 通过。
- 已移除所有临时调试插桩和 Debug Server 相关文件。

## 解决了什么问题

| 问题                                        | 解决方案                                               |
| ------------------------------------------- | ------------------------------------------------------ |
| AI 回复短但 Content Download 耗时 40 秒以上 | SSE 写入后显式 flush，并禁用压缩缓冲                   |

## 踩坑记录

1. **间歇性问题难复现**：本地 dev 环境复现时 SSE 是实时流动的，原始截图的 40 秒延迟可能由上游 LLM 排队、生产环境代理缓冲或首次请求冷启动导致。修复重点放在“消除已知的 SSE 缓冲隐患”。
2. **调试清理要彻底**：清理插桩时漏掉了 `agent-native.ts` 工具调用处的 `dbg(...)`，导致 TypeScript 报 `找不到名称“dbg”`。后续清理需用 `grep` 二次确认无残留。
3. **Node.js HTTP 响应默认行为**：`res.write()` 不保证立即flush；SSE 场景下必须主动处理压缩和 flush。

## “继续生成”按钮无反应问题修复

### 现象

- 助手消息被 `max_tokens` 截断后，UI 显示 ⚠️ 提示 + “继续生成”按钮
- 点击按钮，**完全没反应**：console 无日志、请求未发出、UI 不变化

### 排查思路（假设驱动）

| 假设 | 内容                                                       | 验证结果                                       |
| ---- | ---------------------------------------------------------- | ---------------------------------------------- |
| A    | ChatMessage 按钮 `@click` 没触发 `emit('continue')`        | 模板正确，emit 正常发出                        |
| B    | 父组件 ChatView **没监听** `@continue` 事件                | **命中**：缺 `@continue="handleContinue"` 绑定 |
| C    | handleContinue 引用了错误的 sendMessage                    | 函数实现正确，问题在事件没传上来               |
| D    | sendMessage 函数本身有 bug（比如死锁、isLoading 守卫过严） | 守卫在 push 之前，逻辑正常                     |

### 根因

[ChatView.vue](file:///c:/Users/wdking/Desktop/exam-cram-agent/frontend/src/views/ChatView.vue#L64-L68) 模板中的 `<ChatMessage>` 组件**没有**绑定 `@continue` 事件监听器。子组件 `emit('continue')` 发出后，父组件没接收方，所以“无反应”。

### 修复

| 文件                                      | 改动                                                                              |
| ----------------------------------------- | --------------------------------------------------------------------------------- |
| `frontend/src/views/ChatView.vue`         | `<ChatMessage>` 加 `@continue="handleContinue"` 和 `:is-loading="isLoading"` 绑定 |
| `frontend/src/composables/useChat.ts`     | `sendMessage` 开头加 `if (isLoading.value) return` 守卫，防快速重复点击           |
| `frontend/src/components/ChatMessage.vue` | 按钮加 `:disabled="isLoading"`，文案切换 “继续生成” ⇄ “生成中…”，提供视觉反馈     |

### 验证

- 助手消息被截断时显示 ⚠️ + 按钮
- 点击按钮 → 按钮立即变灰 + 文字 “生成中…”
- 新助手气泡出现，内容流式追加
- 流结束后按钮恢复（若新回答也截断则保持）

## “连一个字都加载不出来” 问题排查

### 现象

- 修复“继续生成”问题后，新发消息**助手气泡完全空白**
- console 显示 `sendMessage` 被调用、收到 `first text chunk`（内容是 `"\n\n"` 2 个换行符）
- 后端 `max_tokens=4096` 已正确配置

### 排查思路（假设驱动）

| 假设 | 内容                                                                            | 验证结果                 |
| ---- | ------------------------------------------------------------------------------- | ------------------------ |
| A    | **HMR 状态错乱**：Vite 推送 useChat 模块更新后，ChatView 解构引用旧 sendMessage | **命中**                 |
| B    | **后端 tsx watch 重载延迟**：进程跑的可能是改之前的代码（旧 max_tokens）        | **疑似**：需手动重启确认 |
| C    | API 端 max_tokens 被拒绝，自动用极小默认值                                      | 未复现，重启后正常       |
| D    | 前端响应式失效（ref 引用错位）                                                  | 未复现，Proxy 包装正常   |

### 取证过程

1. **临时加诊断 log**：
   - 前端 `useChat.ts`：`[DIAG] sendMessage called` / `assistantIdx=` / 每次 text chunk
   - 后端 `agent-native.ts`：`[BACKEND-DIAG] iter=... delta.content len=... finishReason=...`
2. **手动重启后端**（绕过 tsx watch 延迟）：
   ```bash
   taskkill /F /IM node.exe
   npm run dev:server
   ```
3. **硬刷新浏览器**（绕过 HMR 状态残留）：`Ctrl + Shift + R`
4. **重发消息 + 收集日志**

### 根因

**HMR 状态错乱 + 后端进程未真正重启**：

- 改 `useChat.ts` 后，Vite HMR 推送模块更新，但 ChatView 解构的 `sendMessage` 引用旧函数（composable HMR 不会触发调用方 setup 重跑）
- 后端 `tsx watch` 重载有延迟（特别是 Windows 上文件 watcher 不稳定），进程可能跑的仍是改之前的代码，导致 max_tokens 仍为旧值

### 修复

**无需代码修复** —— 靠 “手动重启后端 + 硬刷新浏览器” 绕过：

- 重启后端加载新代码（`max_tokens=4096` + 诊断 log）
- 硬刷新浏览器清掉 HMR 中间态（ref 引用恢复正常）

### 保留的有效代码改进

| 文件                                      | 改动                                       |
| ----------------------------------------- | ------------------------------------------ |
| `frontend/src/views/ChatView.vue`         | `@continue` 事件绑定 + `isLoading` 传递    |
| `frontend/src/composables/useChat.ts`     | `sendMessage` 开头 `isLoading` 守卫        |
| `frontend/src/components/ChatMessage.vue` | 按钮 `:disabled` 状态 + “生成中…” 文字切换 |

### 验证

- 用户反馈 “UI 可以看见内容了”

## 经验总结

1. **HMR 对 composable 不友好**：composable 文件改动后，调用方解构的引用仍指向旧函数。Vite HMR 推送不会自动触发组件 setup 重跑，导致新旧函数混用。
2. **`tsx watch` 在 Windows 上不稳定**：依赖自动重启不靠谱，改后端核心逻辑后**手动重启**（taskkill + npm run dev:server）更稳。
3. **硬刷新是 HMR 状态的清道夫**：`Ctrl + Shift + R` 能清掉所有 HMR 中间态，Vue 组件实例 ref 引用恢复正常。
4. **诊断 log 必须可清理**：每次加诊断代码时打 `[DIAG]` / `[BACKEND-DIAG]` 前缀和 `验证后请删除` 注释，清理时 `grep DIAG` 一次扫完。

## 踩坑记录（下午）

1. **没注意 ChatView 没绑 `@continue`**：之前以为加了，实际没生效（可能 HMR 还原或 git 操作导致丢失）。修复时必须 `grep` 二次确认。
2. **HMR 闭包陷阱**：composable 重载后，旧组件实例仍引用旧 sendMessage 函数，导致 sendMessage 行为与当前代码不一致。`sendMessage` 闭包持有的是 **HMR 时刻的旧 ref**。
3. **`tsx watch` 重载有延迟**：依赖自动重启不靠谱，关键修复后手动重启才稳。
4. **OpenAI 兼容 API 的 max_tokens 行为**：API 端可能有独立的 max_tokens 上限，传过去的 4096 实际可能被降级。**暂时不深究**，靠 max_tokens=4096 + done truncated 字段做容错。
